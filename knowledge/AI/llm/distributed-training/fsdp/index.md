# FSDP

## 核心理解

FSDP（Fully Sharded Data Parallel）是一种分布式训练方法，核心目标是在保持数据并行训练语义的同时，把模型参数、梯度和优化器状态分片存放到多张卡上，从而显著降低单卡显存压力。

直观地说，普通数据并行通常要求每张卡都保留一份完整模型副本。FSDP 则尽量不让每张卡长期持有完整参数，而是把参数拆开分到不同设备上。某个模块真正要计算时，再临时把该模块需要的完整参数聚合出来；计算结束后，再把它重新切回分片状态。

因此，FSDP 的核心不是“让训练更快”本身，而是“让原本单卡或普通 DDP 放不下的大模型能够在多卡上训练”，并在此基础上追求可接受的吞吐和通信成本。

## 工作机制 / 构建过程

FSDP 可以从“参数什么时候完整、什么时候分片”这个角度理解。

假设一个模型被按若干模块包裹在 FSDP 中，每个模块的参数被切分到不同 rank 上。训练过程中通常会发生下面的事：

1. 初始化时，每个 rank 只长期保存自己负责的参数分片，而不是完整参数。
2. 当某个 FSDP 模块即将执行前向计算时，各个 rank 通过通信把该模块参数临时聚合成完整权重。
3. 前向计算完成后，如果配置允许，会尽快释放完整参数，只保留本地分片，以回收显存。
4. 反向传播到该模块时，再次把需要的参数聚合出来，以便计算梯度。
5. 梯度计算完成后，对梯度进行 reduce-scatter，使每个 rank 只保留自己对应的梯度分片。
6. 优化器更新时，每个 rank 更新自己持有的参数分片和相关优化器状态。
7. 整个训练过程中，参数、梯度和优化器状态都以“能分就分”的方式保存在不同设备上。

这套机制的关键收益是：完整模型权重不需要在每张卡上长期常驻。代价则是：前向和反向过程中会插入额外的 all-gather、reduce-scatter 等通信操作。

## 实现方法与工程细节

### FSDP 分片的对象

FSDP 主要分片三类训练状态：

- Parameters：模型参数本身。
- Gradients：反向传播得到的梯度。
- Optimizer States：如 Adam 的一阶矩、二阶矩等优化器状态。

这三类状态里，优化器状态往往非常占显存。FSDP 的价值不仅在于切参数，也在于把优化器状态一起切掉。

### 参数聚合与释放

FSDP 的一个关键工程点是参数的生命周期管理。

在模块真正需要计算之前，参数以分片形式存在；到前向或反向即将计算该模块时，再临时 all-gather 成完整参数；计算结束后尽快释放非本地部分。这样做能把显存峰值压低，但会增加通信频率。

是否能得到好的收益，取决于：

- 模块切分粒度是否合适。
- 通信与计算是否能重叠。
- 网络带宽和拓扑是否足够支撑频繁聚合。
- batch size、序列长度和激活开销是否匹配。

### Auto Wrap Policy

实际使用 FSDP 时，通常不会只在整个模型最外层包一层，而是按模块粒度做包裹。

如果包得太粗：

- 单次 all-gather 的参数量太大。
- 显存峰值可能仍然很高。
- 参数驻留时间更长。

如果包得太细：

- 通信次数会明显增加。
- 调度和元信息开销变重。
- 吞吐量可能下降。

因此工程上常见做法是为 Transformer block 这类稳定结构设置 auto wrap policy，让每个 block 作为一个相对合理的 FSDP 单元。

### Mixed Precision 与 CPU Offload

FSDP 常和 mixed precision 一起使用，以进一步压低显存和通信成本。

常见组合包括：

- 参数或计算使用 `bf16` / `fp16`。
- 梯度同步也使用低精度。
- 在极端显存受限场景下，把部分状态 offload 到 CPU。

但 CPU offload 往往会显著拖慢训练，因此更适合作为“先跑起来”的兜底方案，而不是默认高吞吐配置。

### Checkpoint 与 State Dict

FSDP 下保存和加载 checkpoint 不是普通单卡训练那种“直接存完整模型”思路。

因为参数本身是分片保存的，所以工程上需要明确：

- 保存 full state dict 还是 sharded state dict。
- 恢复训练时是否要求相同 world size。
- 是否要把 optimizer state 一起按分片方式持久化。

这会直接影响训练恢复、跨机器迁移和部署前导出权重的流程设计。

## 与 DDP、ZeRO 的关系

FSDP 和 DDP 都属于数据并行范式，但它们的内存模型差异很大。

DDP 的基本思路是：

- 每张卡有一份完整模型副本。
- 不切参数，只在反向传播时同步梯度。

因此 DDP 实现相对直接，但当模型很大时，参数和优化器状态会很快撑满显存。

FSDP 的思路是：

- 模型状态尽量分片保存。
- 只在需要计算时临时恢复完整参数。

因此它更适合大模型场景，但通信和调试复杂度也更高。

ZeRO 和 FSDP 在目标上非常接近，都是通过分片减少训练状态内存占用。可以粗略理解为：它们都在解决“大模型训练放不下”的问题，只是具体实现路径、框架生态和工程接口不完全一样。很多讨论里也会把 FSDP 看作一类 full sharding 方案在 PyTorch 体系下的代表实现。

## 适用场景与局限

FSDP 更适合以下场景：

- 模型参数规模已经让单卡或 DDP 难以承受。
- 多卡显存总量足够，但单卡显存不足。
- 可以接受更高的通信复杂度来换取可训练性。
- 使用 PyTorch 原生分布式能力，希望减少外部系统依赖。

它的局限通常包括：

- 通信开销明显，网络差时收益会被吞掉。
- wrap 策略、mixed precision、checkpoint 策略都需要仔细调。
- 调试复杂度高于单卡和普通 DDP。
- 对训练脚本、保存恢复流程和监控指标都有更高工程要求。

## 与相关节点的关系

FSDP 属于 `LLM -> Distributed Training` 路径下的训练系统节点。

它和 `Fine-tuning` 有关系，但不是微调方法本身。LoRA 解决的是“训练哪些参数”，FSDP 解决的是“这些参数、梯度和优化器状态如何在多卡上存和算”。两者可以组合使用，例如在大模型微调时既使用 LoRA 降低可训练参数量，也使用 FSDP 管理底座模型和训练状态的显存占用。

它也和 `Optimization` 相关，但不属于优化器范畴。优化器决定参数怎么更新，FSDP 决定这些状态如何分布式存储和同步。

## 待整理

- DDP：完整模型副本式数据并行。
- ZeRO：按训练状态分阶段切分的方案。
- All-Gather：把分片参数临时聚合成完整张量。
- Reduce-Scatter：聚合后再按分片分发结果。
- Auto Wrap Policy：决定哪些模块单独包 FSDP。
- CPU Offload：把部分训练状态放到 CPU。
- State Dict：FSDP 下保存和恢复模型状态的格式。
- Sharding Strategy：定义参数和状态如何切分。
