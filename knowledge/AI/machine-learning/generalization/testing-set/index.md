# Testing Set

## 核心理解

Testing set 是在模型训练完成后，用来评估模型泛化能力的数据集合。

它的核心职责不是参与训练，而是尽可能模拟模型未来在未见样本上的真实表现。因此，testing set 上得到的 testing error，通常被用来近似 generalization error。

## 为什么需要 Testing Set

如果只在 training set 上评估模型，那么看到的只是 training error。

但 training error 只能说明模型对已见数据拟合得怎么样，不能直接说明它对新样本是否也有效。为了更接近真实部署场景，评估时需要保留一部分未参与训练的数据，这就是 testing set 的意义。

## Testing Set 的角色

可以粗略理解为：

- training set 用于学习参数。
- validation set 用于调参、模型选择和 early stopping。
- testing set 用于最终评估。

因此，testing set 更像是一种“最后检验”，而不是训练过程中的反复参考对象。

## 一个合格 Testing Set 应满足什么

### 与训练过程隔离

testing set 不应参与参数更新，也不应被用来直接指导模型设计。

如果测试集反复被拿来调超参数、选模型或决定训练策略，那么它实际上就不再是纯粹的测试集，而是部分退化成了验证集。

### 尽可能代表真实分布

测试集之所以有价值，是因为它试图模拟未来会遇到的真实样本分布。

因此：

- 测试集应尽可能与训练集互斥。
- 测试集应尽可能保持与真实任务一致的数据分布。
- 如果是分类任务，通常应注意类别比例是否失衡，必要时使用分层采样。

### 样本量要足够

如果测试集太小，评估结果会有较大波动，容易被少量样本主导。

因此，测试集既要保持独立性，也要有足够规模，才能让评估结果更稳定。

## Testing Error 与 Generalization Error 的关系

testing error 通常被看作 generalization error 的一个近似，而不是它本身。

两者越接近，通常越依赖以下条件：

- 测试集没有泄漏到训练过程。
- 测试集能够代表真实分布。
- 评估指标与真实任务目标一致。
- 测试集规模足够大。

如果这些条件不满足，那么 testing error 就可能对 generalization error 形成失真估计。

## 常见误用

### 用测试集调参

这是最常见的问题之一。

如果每次模型效果不好，就根据测试集结果继续调学习率、改结构、选特征，那么测试集信息实际上已经被训练过程吸收。这样得到的测试结果会偏乐观，不能再视为真正独立的泛化评估。

### 测试集分布不代表真实任务

如果测试集过于理想化，或者采样方式与真实部署环境差别很大，那么即使测试指标很好，也不意味着模型在真实场景中一定有效。

### 测试集过小

测试集过小时，评估结果的随机波动会明显增加，模型之间的比较也可能变得不可靠。

## 与 Evaluation Methods 的关系

不同评估方法会以不同方式构造 testing set：

- Holdout 中，测试集是一次划分后直接保留下来的那部分数据。
- K-fold cross validation 中，每一折都会轮流充当测试集。
- Leave-One-Out 中，每次只有一个样本作为测试集。
- Bootstrapping 中，没有被采入训练集的样本可用于包外估计。

因此，Evaluation Methods 讨论的是“如何构造测试集”，而 Testing Set 讨论的是“测试集在整个学习流程中应承担什么角色”。

## 与相关节点的关系

- `Generalization` 解释为什么需要通过测试集近似评估泛化误差。
- `Evaluation Methods` 解释如何构造测试集。
- `Validation Set` 负责训练过程中的调参和选择，因此 testing set 不应承担这类职责。
