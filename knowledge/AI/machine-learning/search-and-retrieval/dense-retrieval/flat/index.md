# Flat

## 核心理解

Flat 是一种最直接的向量检索方式：不使用近似索引结构，不预先构建分层图、聚类中心或压缩码本，而是在查询时直接把 query vector 和库中的所有 document vectors 逐个计算相似度，再取 Top K。

所以 Flat 的本质不是“更聪明地缩小搜索范围”，而是“完整扫描全部候选，再按真实相似度排序”。如果向量数量不大，或者系统更关心精确性而不是极致吞吐，Flat 往往是最稳定、最容易理解的基线方案。

## 工作机制

Flat 的检索流程非常直接：

```text
document vectors -> sequential scan / batch similarity compute -> top k
```

查询阶段通常包含三步：

1. 把用户查询编码成 query vector。
2. 让 query vector 与全部 document vectors 计算相似度。
3. 选出分数最高的 Top K 结果返回。

如果使用 cosine similarity，系统通常会先对向量做归一化，再计算内积；如果使用 dot product 或 L2 distance，则直接按对应度量计算。

从结构上看，Flat 没有复杂索引导航过程。它更像一个“全量比较器”，而不是 ANN 那种“近似搜索加速结构”。

## 为什么它叫 Flat

叫 Flat，强调的是它没有额外的层次化或分区化索引结构。

和 HNSW、IVF 这类方法相比，Flat 不会：

- 构建多层图。
- 先把向量划入若干聚类桶。
- 用量化方法先压缩向量再近似比较。

它直接保存原始向量，并在查询时对整个向量集合做扫描。因此它常被看作 exact search 的代表实现。

## 和 ANN 的区别

Flat 通常对应精确最近邻搜索，而不是近似最近邻搜索。

区别可以概括为：

- Flat：扫描全部向量，结果最接近“真实 Top K”。
- ANN：只扫描部分候选或通过近似结构加速，速度更快，但可能漏掉真正最近的向量。

这意味着 Flat 的主要优点是结果稳定、实现简单、评测清晰；主要代价是数据规模上来后，查询延迟和计算成本会上升得很快。

所以在工程里，Flat 经常被拿来做两件事：

- 作为小规模系统的直接检索方案。
- 作为 ANN 系统的精确基线，用来衡量近似索引到底损失了多少召回。

## 工程实现要点

### 1. 向量存储形式

Flat 通常直接存储完整浮点向量，例如 `float32` 或 `float16`。

存储成本大致和下面这个量成正比：

```text
向量数 × 向量维度 × 每维字节数
```

例如 100 万条、每条 768 维、`float32` 存储的向量，体量就已经比较可观。

Flat 并不要求把全部向量都放在 GPU。GPU 只是加速相似度计算的一种实现手段；在很多系统里，向量也可以存放在 CPU 内存、磁盘或分层缓存中，具体取决于数据规模、延迟要求和成本预算。

### 2. 计算方式

虽然概念上是“逐个比较”，但真实系统不会真的写成低效 for-loop。

常见实现会使用：

- 矩阵乘法批量计算相似度。
- SIMD 指令加速。
- GPU 并行计算。
- 分块扫描，避免一次性读入过多内存。

所以 Flat 的慢，通常不是“实现粗糙”，而是它必须面对全量比较这件事本身的复杂度。

### 3. 时间复杂度特点

如果库里有 \(N\) 个向量，每次查询都要与全部候选比较一次，那么检索成本通常随 \(N\) 线性增长。

这也是 Flat 最大的扩展瓶颈：

- 向量库越大，单次查询越慢。
- QPS 越高，计算资源压力越大。
- 如果需要低延迟在线服务，规模一大就很难只靠 Flat 支撑。

### 4. 结果精确性

Flat 的一个关键价值在于，它不引入 ANN 那种“索引近似误差”。

只要：

- query encoder 和 document encoder 一致，
- 相似度函数设置正确，
- 向量数据没有损坏，

那么 Flat 返回的就是在当前相似度定义下最真实的 Top K 结果。

因此在调试 Dense Retrieval 系统时，Flat 很适合作为基准参照物。

## 什么时候适合用 Flat

适合：

- 数据规模还不大。
- 更关心精确结果，而不是极限速度。
- 需要搭建 Dense Retrieval 原型或离线评测基线。
- 需要验证 embedding 模型是否真的有效。
- 需要和 ANN 方法做召回率对比。

不适合直接单独承担大规模在线检索：

- 向量库很大。
- 查询延迟要求很低。
- QPS 很高。
- 成本预算敏感。

这时通常会转向 HNSW、IVF、PQ 等近似索引或压缩方案。

## 在工程中的典型角色

在真实系统里，Flat 经常不是最终形态，而是下面几类角色之一：

- 原型阶段的第一版向量检索实现。
- 小规模内部知识库的直接方案。
- ANN 系统的评测基线。
- 离线重排前的候选生成对照组。
- GPU 检索场景下的精确相似度计算组件。

也就是说，Flat 很重要，但它的重要性常常来自“基线”和“参照物”，而不只是“最终生产方案”。

## 与相关节点的关系

- Dense Retrieval：Flat 是 Dense Retrieval 中最基础的向量检索实现之一。
- Embedding：Flat 不负责生成向量，只负责在已有向量之间做精确相似度搜索。
- ANN：Flat 可以作为 ANN 的对照基线，帮助判断近似索引损失了多少真实召回。
- HNSW / IVF / PQ：这些方法都试图在 Flat 的全量精确比较之外，用更低延迟换取可接受的近似误差。
- Vector Database：很多向量数据库都会提供 Flat 作为一种索引选项或基线实现。
