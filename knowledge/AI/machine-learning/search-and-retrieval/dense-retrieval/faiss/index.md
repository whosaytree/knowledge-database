# FAISS

## 核心理解

FAISS 是 Meta 开源的向量相似度搜索库，名字来自 Facebook AI Similarity Search。

它解决的问题不是“如何把文本变成向量”，而是“当我们已经有大量向量后，怎样高效地找到最相近的那些向量”。

在 Dense Retrieval 系统里，FAISS 通常承担的是向量索引与近邻搜索这一层能力。Embedding 模型负责生成向量，FAISS 负责在这些向量上执行 Top K 近邻检索。

## 它在 Dense Retrieval 流程中的位置

Dense Retrieval 的离线与在线流程可以简化为：

```text
documents -> chunking -> embeddings -> vector index
query -> query embedding -> nearest neighbor search -> top k results
```

FAISS 主要位于后半段：

- 离线阶段：接收 document vectors，构建向量索引。
- 在线阶段：接收 query vector，执行最近邻搜索。
- 输出阶段：返回最相近的向量 id 和距离分数。

因此，FAISS 关注的是“搜索效率与索引结构”，不是“语义表示学习”。

## FAISS 处理的对象

FAISS 的基本输入是向量矩阵。

例如有很多 document embeddings：

```text
doc 1 -> [0.12, -0.03, 0.88, ...]
doc 2 -> [0.19,  0.07, 0.63, ...]
doc 3 -> [-0.02, 0.44, 0.51, ...]
```

系统会把这些向量写入 FAISS 索引中。查询时，再把用户查询编码成 query vector：

```text
query -> [0.10, 0.01, 0.79, ...]
```

然后在索引中搜索与它最近的 Top K 向量。

返回结果通常不是原始文本，而是：

- 向量 id
- 相似度或距离
- 可选的额外分数信息

业务系统再根据这些 id 回查 chunk、文档和元数据。

## FAISS 为什么重要

如果只有几千个向量，可以直接暴力计算 query 和所有 document vectors 的相似度。

但当向量规模上升到几十万、几百万甚至更多时，逐个比较的代价会很高。FAISS 的价值就在于，它提供了一套面向大规模向量检索的高效实现，帮助系统在速度、内存、精度之间做折中。

它的重要性主要体现在几个方面：

- 提供精确搜索和近似搜索两类索引。
- 支持多种距离度量，例如 L2 和 Inner Product。
- 支持 CPU 和 GPU 加速。
- 支持向量压缩与分片，适合大规模检索。
- 被大量检索系统、向量数据库和 RAG 系统作为底层能力使用。

## FAISS index 大小取决于什么

FAISS index 会不会很大，取决于：

- 向量数量
- 向量维度
- 每维使用的数据类型，例如 `float32`
- 索引结构本身是否需要额外元数据

最朴素的 `IndexFlat` 基本可以理解为“把所有向量原样存下来”，所以体积大致接近：

$$
\text{index size} \approx \text{num vectors} \times \text{dimension} \times \text{bytes per dimension}
$$

例如有 100 万条 768 维向量，使用 `float32`：

$$
1{,}000{,}000 \times 768 \times 4 \approx 3.1 \text{ GB}
$$

这时索引本身就已经是 GB 级别，还不包括业务侧常见的：

- id 映射表
- 原始文本存储
- 元数据存储

如果使用 `IVF`、`PQ` 或它们的组合，索引体积通常会小很多，因为它们不会简单保存完整原始向量，而是引入分桶、量化或压缩表示。但这种缩减会带来额外训练步骤，以及一定的近似误差。

## 和普通 index 相比，多了什么

如果把 FAISS 和传统数据库索引、B-Tree 或倒排索引相比，差异不只是“换了一种存法”，而是服务的检索问题已经不同了。

传统索引通常处理的是：

- 精确键查找
- 范围查找
- 词项到文档列表的映射

FAISS 处理的是：

- 向量空间中的最近邻搜索
- 相似度排序
- 近似最近邻加速

因此它常常会比“普通 index”多出一些向量检索特有的结构，例如：

- 原始向量或压缩后的向量表示
- 聚类中心，例如 IVF 中的 coarse centroids
- 量化码本，例如 PQ 中的 codebooks
- 图结构邻接信息，例如 HNSW 中的邻居连接
- 用于搜索过程的辅助元数据和参数

不过如果比较对象是最基础的 `IndexFlat`，那么它并没有引入特别复杂的索引结构，本质上更接近“向量集合 + 相似度搜索接口”。真正会明显“多出很多东西”的，通常是 `IVF`、`PQ`、`HNSW` 这类 ANN 索引。

## FAISS 的两类基本搜索思路

### 1. 精确搜索

最直接的方法是保存所有向量，然后对查询向量和每个文档向量逐一计算距离。

这种方法结果最准确，因为不会漏掉真正最近的向量，但成本高，向量规模大时延迟明显上升。

在 FAISS 中，`IndexFlatL2`、`IndexFlatIP` 属于这一类思路。

它们更适合：

- 数据规模还不大。
- 需要基线实验或离线评测。
- 希望先验证 embedding 效果，再考虑 ANN 优化。

### 2. 近似最近邻搜索

当向量规模变大时，系统通常不会遍历全部向量，而是使用近似方法缩小搜索范围。

这类方法不一定总能找出“绝对最近”的向量，但能显著提升速度，并把召回损失控制在可接受范围内。

FAISS 中常见的近似索引包括：

- IVF：先把向量分配到聚类桶里，查询时只搜索部分桶。
- PQ：对向量做压缩编码，降低存储和计算成本。
- HNSW：用图结构组织近邻关系，加速搜索。
- IVF-PQ：先分桶，再压缩，是常见的大规模方案。

## 距离度量与向量归一化

FAISS 本身只负责做向量空间里的近邻搜索，因此距离定义非常重要。

常见做法包括：

- L2 Distance：欧氏距离，适合直接比较几何距离。
- Inner Product：内积，常用于最大内积搜索。
- Cosine Similarity：余弦相似度通常通过“先归一化向量，再做内积”来实现。

工程上一个常见误区是：embedding 模型按 cosine similarity 使用，但写入 FAISS 时没有做归一化，或者索引类型和打分方式不匹配。这样会导致检索结果偏移。

所以 FAISS 的索引选择，必须和 embedding 模型的训练目标、相似度定义一起看。

## 一个典型工作流程

在知识库检索或 RAG 系统中，FAISS 的使用流程通常是：

### 离线构建

```text
documents
  -> chunking
  -> embedding model
  -> document vectors
  -> build FAISS index
  -> save index + id mapping
```

这里除了索引本身，还需要维护一份 `vector id -> chunk / document` 的映射关系。

### 在线查询

```text
user query
  -> query embedding
  -> search in FAISS
  -> top k vector ids
  -> lookup text chunks
  -> rerank / filter / send to LLM
```

也就是说，FAISS 只负责“从向量到候选 id”，完整系统还需要：

- 文本存储
- 元数据过滤
- 权限控制
- 结果重排
- 上下文拼接

## 哪些东西必须放在 GPU 上

严格来说，没有哪些部分是“必须”放在 GPU 上的。FAISS 的很多能力可以完全在 CPU 上运行，尤其是在数据规模不大或吞吐要求不高时。

更准确的说法是：有些环节更适合放在 GPU 上，而不是必须依赖 GPU。

通常更适合 GPU 的包括：

- 大批量向量写入前的索引构建
- 大规模批量查询
- 计算量很高的相似度搜索
- IVF、PQ 等需要训练聚类或量化参数的过程

通常不一定需要 GPU 的包括：

- 小到中等规模索引
- 本地实验或开发阶段验证
- 查询量不高的服务
- 结构较简单的 `IndexFlat` 或小型 ANN 索引

工程上常见的实际分工是：

- Embedding 模型生成向量时更常依赖 GPU。
- FAISS 可以跑在 CPU，也可以迁到 GPU 加速。
- 是否上 GPU，更多取决于向量规模、QPS、延迟目标和成本预算。

所以问题不应理解成“哪些模块必须用 GPU”，而应理解成“哪些阶段在当前规模下值得用 GPU”。

## FAISS 与向量数据库的关系

FAISS 不是完整的向量数据库。

它更接近“高性能向量索引与搜索引擎库”，核心价值是索引结构和近邻搜索算法实现。它通常不直接提供完整的生产系统能力，例如：

- 多租户管理
- 权限过滤
- 分布式副本
- 自动扩缩容
- 文档级事务语义
- 完整的在线服务接口

因此可以这样理解：

```text
FAISS 解决的是“怎么高效做向量近邻搜索”
向量数据库解决的是“怎么把向量检索作为长期稳定系统运行”
```

很多向量数据库或检索系统，底层就会使用 FAISS 或借鉴它的索引思路。

## 工程实现要点

在实际系统里，FAISS 相关问题通常不在“能不能跑起来”，而在“效果、延迟、成本如何平衡”。

常见关注点包括：

- 索引类型选择：Flat、IVF、PQ、HNSW 各有不同权衡。
- 训练成本：某些索引如 IVF、PQ 需要先训练。
- 向量归一化：必须和相似度定义匹配。
- 增量更新：新增、删除、重建索引的流程要提前设计。
- id 映射：FAISS 返回的是向量 id，不是业务对象本身。
- 内存占用：向量维度、数据规模和索引结构会直接影响成本。
- GPU 使用：大规模离线建索引或高吞吐查询时，GPU 可能明显加速。
- 评测方式：要同时看召回率、延迟、吞吐和资源消耗，而不是只看单一指标。

## 什么时候优先考虑 FAISS

适合：

- 已经有 embedding，准备自己构建向量检索能力。
- 需要本地或自托管的高性能向量搜索。
- 想清楚控制索引结构、参数和性能调优。
- 做 Dense Retrieval、RAG、相似文本检索或推荐召回实验。

不适合单独承担全部系统职责：

- 需要完整的多租户、权限、分布式和运维能力。
- 需要复杂元数据过滤和稳定在线服务接口。
- 团队更希望直接使用成熟向量数据库，而不是自己补整套服务层。
