# BM25

## 核心理解

BM25 是关键词检索中最常用的相关性打分函数之一，通常作为 Sparse Retrieval 的强基线。

它的目标是回答一个问题：对于某个查询，一篇文档和这个查询有多相关？

BM25 可以理解为 TF-IDF 的工程化改进。它不仅考虑查询词在文档中是否出现、出现多少次、在整个语料库中是否稀有，还会控制两个关键问题：

- 词频收益递减：同一个词出现更多次通常更相关，但不是线性增加。
- 文档长度归一化：长文档天然更容易命中更多词，需要避免被不公平地抬高分数。

因此，BM25 常用于搜索引擎、知识库检索、RAG 系统中的关键词召回，以及 Hybrid Retrieval 的 sparse 分支。

## 一个简单例子

假设用户查询：

```text
sparse retrieval
```

有两篇候选文档：

```text
doc_1: sparse retrieval
doc_2: sparse retrieval retrieval retrieval retrieval
```

如果只看词频，`doc_2` 可能会因为 `retrieval` 出现很多次而得到过高分数。

BM25 不会让词频无限线性增长。一个词从出现 0 次到出现 1 次非常重要，从 1 次到 2 次也可能有帮助，但从 10 次到 11 次通常不会让相关性大幅提升。

这就是 BM25 的词频饱和思想。

再看另一个情况：

```text
doc_3: sparse retrieval
doc_4: sparse retrieval + 很长的无关正文
```

如果两篇文档都命中了查询词，BM25 会考虑文档长度。较长文档因为包含更多词，本来就更容易命中查询，因此需要进行长度归一化。

## 打分公式

BM25 的常见形式是：

$$
\mathrm{score}(D, Q) = \sum_{q_i \in Q} \mathrm{IDF}(q_i)
\cdot
\frac{f(q_i, D) \cdot (k_1 + 1)}
{f(q_i, D) + k_1 \cdot \left(1 - b + b \cdot \frac{|D|}{\mathrm{avgdl}}\right)}
$$

其中：

- `D`：当前文档。
- `Q`：查询。
- `q_i`：查询中的第 i 个词项。
- `f(q_i, D)`：词项 `q_i` 在文档 `D` 中出现的次数。
- `IDF(q_i)`：词项 `q_i` 的逆文档频率。
- `|D|`：文档长度。
- `avgdl`：语料库中的平均文档长度。
- `k1`：控制词频饱和速度。
- `b`：控制文档长度归一化强度。

## 关键组成

### 1. IDF：词项稀有程度

IDF 衡量一个词在整个语料库中有多稀有。

如果一个词出现在很多文档里，例如 `the`、`is`、`系统`，它对区分文档的帮助较小。

如果一个词只出现在少数文档里，例如产品型号、错误码、函数名、论文术语，它通常更能说明文档和查询相关。

BM25 会给稀有词更高权重，给高频常见词更低权重。

### 2. TF Saturation：词频收益递减

词频越高，文档通常越可能相关。

但这种相关性不会无限线性增长。一个查询词出现 20 次的文档，不一定比出现 10 次的文档相关两倍。

BM25 使用 `k1` 控制词频增长曲线：

- `k1` 越大，词频增长带来的影响越明显。
- `k1` 越小，词频更快进入饱和状态。

常见默认值大约在 `1.2` 到 `2.0` 之间。

### 3. Length Normalization：文档长度归一化

长文档包含更多词，因此更容易命中查询词。

BM25 使用 `|D| / avgdl` 比较当前文档长度和平均文档长度，并通过参数 `b` 控制长度惩罚强度：

- `b = 0`：不考虑文档长度。
- `b = 1`：完整使用文档长度归一化。
- 常见默认值是 `0.75`。

这使 BM25 能在短文本和长文本之间取得相对平衡。

## 和倒排索引的关系

BM25 通常运行在倒排索引之上。

倒排索引负责快速找到包含查询词的候选文档，并提供打分所需的信息，例如：

- 查询词出现在哪些文档中。
- 查询词在每篇文档中的词频。
- 每篇文档的长度。
- 每个词项的文档频率。
- 整个语料库的文档总数和平均文档长度。

因此，倒排索引解决“去哪找候选文档”，BM25 解决“候选文档如何排序”。

## 工程实现要点

在真实系统中，BM25 通常不会在查询时从原始文档重新统计信息，而是依赖索引阶段预先保存的统计量。

常见实现会保存：

```text
posting list:
  term -> [
    { doc_id, term_frequency }
  ]

document stats:
  doc_id -> document_length

collection stats:
  total_document_count
  average_document_length
  document_frequency_by_term
```

查询时，系统会：

1. 对查询进行分词和归一化。
2. 从倒排索引中取出每个查询词的 posting list。
3. 读取词频、文档长度和语料统计信息。
4. 对每个候选文档累加 BM25 分数。
5. 按分数排序返回 Top K 结果。

## 参数影响

BM25 最常见的两个参数是 `k1` 和 `b`。

`k1` 影响词频饱和：

- 较小的 `k1`：词频增加很快饱和，适合避免重复词过度影响排序。
- 较大的 `k1`：词频影响更明显，适合词频确实有较强相关性的场景。

`b` 影响文档长度归一化：

- 较小的 `b`：弱化长度惩罚。
- 较大的 `b`：强化长度惩罚。
- 对字段长度差异很大的语料，需要更谨慎调整。

默认参数通常已经足够强，很多系统会先使用默认 BM25，再根据评测集调参。

## 常见用途

BM25 常用于：

- 搜索引擎中的关键词排序。
- 文档知识库检索。
- 企业内部搜索。
- 日志、代码、商品、论文等垂直搜索。
- RAG 系统中的第一阶段召回。
- Hybrid Retrieval 中的 sparse 分支。
- Dense Retrieval 或 reranker 的候选生成阶段。

它特别适合处理专有名词、术语、编号、错误码、函数名、文件名和实体名等精确匹配需求。

## 局限性

BM25 主要依赖词项匹配，不理解深层语义。

如果用户查询和文档表达不同，但意思相近，BM25 可能无法召回。例如查询 `car repair` 时，只包含 `automobile maintenance` 的文档可能得不到高分。

BM25 也不天然理解上下文、否定关系、复杂意图和多跳推理。

因此，在现代检索系统中，BM25 经常和 Dense Retrieval、查询改写、同义词扩展、Learning to Rank 或 reranker 结合使用。
