# 组织规则

## 文件与目录

- `meta/` 只存放项目说明、协作方式、组织规则等元信息。
- `knowledge/` 只存放真实知识节点与知识结构。
- `catalog.md` 作为全局入口，不承载长篇知识内容。
- `README.md` 说明项目用途、结构和当前状态。

## 节点原则

后续添加知识节点时，默认采用以下原则：

- 一个知识节点对应一个稳定入口。
- 对于可能继续扩展的主题，优先采用“目录 + index.md”的节点结构。
- 目录名使用英文小写和连字符，保持 URL / 路径稳定，例如 `inverted-index/`。
- 文件标题使用知识节点的正式显示名，例如 `# Inverted Index`。
- 节点内容优先解释概念本身、工作机制、构建过程和实现细节，避免泛泛罗列用途和局限。
- 当一个节点开始承载多个独立主题时，优先考虑拆分为子节点。

## 节点结构约定

对于可扩展主题，默认使用：

```text
node-name/
  index.md
```

例如：

```text
search-and-retrieval/
  index.md
  sparse-retrieval/
    index.md
    inverted-index/
      index.md
```

这样做的好处是：

- 当前节点有稳定入口。
- 后续可以自然添加子节点。
- 目录结构能表达知识层级。
- 文件名不需要承载全部语义，正式名称由 Markdown 标题表达。

## 父节点与比较内容

父节点除了提供子节点入口，也可以承载少量稳定的横向比较内容，用于帮助理解同一主题下多个方法之间的差异。

这类内容适合放在父节点中的前提是：

- 比较对象属于同一个稳定主题域。
- 比较内容会随着子节点扩展而持续复用。
- 该内容服务于方法选择、结构理解或知识导航，而不是一次性问题。

例如在 `Optimization` 节点中，可以维护 `SGD`、`Adam`、`Momentum`、`RMSProp` 等方法的比较框架。

相反，以下内容不宜单独建节点：

- 只对应两个节点之间的临时对比关系。
- 不是稳定知识对象的 `x-vs-y` 标题。
- 很可能随着后续扩展被并入更大父主题的局部比较。

## 子节点排序原则

父节点中的子节点顺序默认不强制按字母排序。

当存在明显的知识依赖、抽象层级或阅读顺序时，应优先按照理解路径排序。例如：

- 基础方法排在扩展方法前。
- 母体概念排在变体前。
- 更适合作为入门入口的节点排在更复杂的节点前。

例如在 `Optimization` 中，`SGD` 应排在 `Adam` 前，因为后者可以视为在前者基础上的进一步扩展。

## 公式表达约定

当知识节点涉及数学公式、推导或符号表达时，正文优先使用 LaTeX 公式块或行内公式。

默认要求：

- 可渲染的数学表达写在公式语法中。
- 不要用代码块代替公式展示。
- 代码块只用于代码、伪代码、命令、目录树或纯文本格式示例。
- 如果一个概念强依赖公式理解，应先保证公式可读，再补文字解释。

## 目录同步规则

新增、移动、重命名或删除知识节点时，必须同步更新相关目录入口：

- 父节点的 `index.md` 应包含直接子节点入口。
- `catalog.md` 应作为全局目录同步更新。
- 如果变更影响多层路径，应逐层检查上级节点是否需要补充或调整链接。
- 目录入口应反映真实文件层级，不使用临时分类或虚拟路径。

## 暂定节点模板

后续新增节点时，可以按需使用这个结构：

```markdown
# 节点标题

## 核心理解

## 工作机制 / 构建过程

## 实现方法与工程细节

## 与相关节点的关系

## 子节点

- [子节点标题](child-node/index.md)

## 待整理

- 术语 A：20 字以内说明它是什么。
- 术语 B：20 字以内说明它是什么。
```

这个模板不是强制格式，具体结构应根据知识节点类型调整。

其中：

- `工作机制 / 构建过程` 是主体部分，应展开说明。
- `实现方法与工程细节` 用于记录数据结构、算法步骤、存储形式、复杂度、系统实现等内容。
- `与相关节点的关系` 用于说明当前节点和父节点、兄弟节点、上下游技术之间的关系。
- `子节点` 只在当前节点已有直接子节点时保留。
- `待整理` 中每个条目应附带 20 字以内的简短说明，避免只列孤立名词。
